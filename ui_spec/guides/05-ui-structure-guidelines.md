# 05. UI Structure Guidelines

`ui_spec`를 생성할 때 참고하는 화면 구조용 UI 가이드입니다.

이 문서는 코드 구조, Flutter 폴더 구조, 라우터 설계 문서가 아닙니다.
목적은 에이전트가 일관된 화면 구성을 만들 수 있도록
모바일 앱의 기본적인 UI 해석 기준을 제공하는 것입니다.

## 목적

- 화면 역할이 비슷하면 비슷한 UI 밀도와 위계를 유지
- 앱 전반의 공통 UI 요소를 반복 가능한 패턴으로 정리
- `ui_spec` 생성 시 화면 하나만 예쁘게 만드는 것이 아니라
  앱 전체가 이어지는 느낌을 갖도록 기준 제공

## 기본 원칙

- 한 화면에는 핵심 목적을 하나만 둔다.
- 첫 화면은 설명보다 방향 제시와 시작 행동이 더 중요하다.
- 목록 화면은 탐색과 상태 파악이 빨라야 한다.
- 상세 화면은 읽기 흐름과 주요 액션의 우선순위가 분명해야 한다.
- 설정 화면은 장식보다 옵션의 의미와 현재 상태 표시가 중요하다.
- 공통 UI는 화면마다 새로 발명하지 말고 반복 가능한 패턴으로 맞춘다.

## 권장 화면 세트

`todo` 같은 생산성 앱의 기본 세트는 아래 구성을 우선 고려한다.

1. Intro screen
2. Home or list screen
3. Detail screen
4. Settings screen
5. Optional modal, empty, loading, done state

모든 앱이 이 구조를 강제로 따라야 하는 것은 아니지만,
별도 요구가 없으면 위 흐름을 기본 템플릿으로 삼는다.

## Intro Screen

- 앱의 성격, 핵심 가치, 시작 행동을 한 화면에서 보여준다.
- 보통 요소는 `브랜드/라벨`, `큰 제목`, `짧은 설명`, `대표 비주얼`, `시작 버튼` 조합을 우선한다.
- 첫 화면에서는 옵션을 너무 많이 주지 않는다.
- CTA는 1개를 우선하고, 보조 행동이 필요해도 시각적 우선순위는 낮춘다.
- 대표 비주얼은 실제 앱 사용 장면을 축약해서 보여주는 카드/목록/체크 패턴이 적합하다.
- 제목이나 설명이 2줄 이상이 될 수 있으면, 아래 대표 카드와 최소 안전 간격을 먼저 확보한 뒤 내부 요소를 배치한다.
- 대표 카드 안에서도 제목, 요약 카드, 리스트 행이 서로 덮지 않도록 텍스트 블록 높이를 보수적으로 잡는다.

## Home Or List Screen

- 사용자가 앱에 들어온 뒤 가장 자주 보는 중심 화면으로 본다.
- 상단에는 현재 날짜, 화면 제목, 빠른 액션, 상태 요약 중 필요한 것만 배치한다.
- 본문은 목록, 섹션 카드, 필터, 진행 상태 등 실제 사용 흐름을 보여줘야 한다.
- `todo` 앱이라면 `오늘 할 일`, `중요`, `완료`, `루틴` 같은 묶음을 먼저 고려한다.
- 카드와 리스트 아이템은 정보 위계가 명확해야 하며,
  제목, 보조 설명, 상태값, 진입 액션을 지나치게 섞지 않는다.

## Detail Screen

- 하나의 항목이나 프로젝트를 깊게 보는 화면으로 해석한다.
- 화면 상단에는 제목과 back 맥락이 분명해야 한다.
- 본문은 설명, 하위 작업, 일정, 메모, 첨부 성격의 정보로 구분한다.
- 완료, 수정, 삭제처럼 위험도나 중요도가 다른 액션은 구분해서 배치한다.
- 상세 화면도 정보량이 많아질수록 섹션 구분이 선명해야 한다.

## Settings Screen

- 설정 화면은 앱의 보조 화면이다.
- 과한 히어로 비주얼보다 명확한 라벨, 현재값, 변경 가능성 표현이 우선이다.
- 보통 `프로필/계정`, `앱 설정`, `알림`, `언어`, `정보` 같은 그룹 구조를 사용한다.
- 각 행은 `라벨`, `현재값 또는 설명`, `이동/변경 affordance` 조합을 기본으로 본다.
- 토글, 선택, 링크 이동은 한 화면 안에서 구분 가능해야 한다.
- destructive action이 있다면 일반 설정 항목과 시각적으로 분리한다.
- `todo` 앱이라면 최소한 `알림`, `정렬 방식`, `기본 보기`, `언어`, `앱 정보` 정도를 기본 후보로 본다.

## App Bar

- `app bar`는 장식용 띠가 아니라 현재 맥락을 보여주는 상단 구조로 해석한다.
- 기본적으로 `제목`, `뒤로가기 또는 메뉴`, `보조 액션` 정도만 둔다.
- 텍스트가 길어지면 제목 1개와 보조 설명 1개까지만 허용하는 방향을 우선한다.
- 화면마다 높이, 좌우 여백, 제목 크기가 크게 흔들리지 않게 맞춘다.
- 홈 화면의 `app bar`는 현재 위치와 주요 진입 버튼을,
  상세 화면의 `app bar`는 back 맥락과 현재 항목 이름을 우선한다.
- 설정 화면의 `app bar`는 단순하고 안정적인 구성이 적합하다.

## End Drawer

- `endDrawer`는 보조 탐색과 보조 설정 진입 UI로 본다.
- 메인 작업을 방해하지 않게 정보량을 제한한다.
- 기본적으로 상단 요약 영역, 메뉴 리스트, 하단 보조 액션 정도의 3구조를 권장한다.
- 상단에는 앱 이름, 사용자 상태, 짧은 설명 또는 현재 필터 상태를 둘 수 있다.
- 메뉴 리스트에는 `설정`, `보관함`, `완료한 일`, `도움말` 같은 보조 이동 항목을 배치한다.
- 하단에는 로그아웃, 버전 정보, 정책 링크처럼 낮은 빈도의 요소를 둔다.
- drawer 안에서도 현재 선택 위치가 드러나야 한다.
- drawer 항목은 홈 본문 액션과 경쟁하지 않게 차분한 밀도로 설계한다.

## Common UI Patterns

- 버튼은 primary, secondary, destructive 정도의 계층만 우선 정의한다.
- 카드 모양, 라운드 값, 그림자 강도는 앱 전체에서 제한된 패턴으로 반복한다.
- 하단 CTA나 주요 액션 버튼의 입체감은 가능하면 버튼 자체의 `DROP_SHADOW`와 fill 대비로 해결하고, 받침 패널을 하나 더 깔아 버튼을 이중 구조로 만드는 방식은 꼭 필요할 때만 사용한다.
- 목록 행 높이, 아이콘 크기, 텍스트 위계는 화면마다 크게 바뀌지 않게 유지한다.
- empty state는 단순 안내 문구 + 유도 행동 1개 조합을 우선한다.
- loading state는 실제 최종 레이아웃과 비슷한 skeleton 또는 간단한 진행 상태로 맞춘다.
- 텍스트 블록 아래에 다른 블록 요소가 오면, 시안상 간격이 촘촘해 보여도 실제 렌더 기준 안전 여백을 우선한다.
- 긴 한글 문구는 영문보다 줄바꿈이 빨라질 수 있으므로 카드 내부 제목 폭과 아래 요소 간격을 더 보수적으로 잡는다.

## Selection And Field Patterns

- 필터 칩, segmented control, tab strip처럼 상호배타 선택 UI는 단순 CTA 버튼 묶음이 아니라 선택 그룹으로 해석한다.
- 선택용 카드와 상세 진입용 row는 같은 카드 모양이어도 주행동이 다를 수 있으므로 시각 패턴만 보고 동일 affordance로 취급하지 않는다.
- 선택용 카드에는 현재 선택 상태, 기본값, 해제 affordance가 보이면 구조적으로도 분리 가능한지 먼저 검토한다.
- 검색창이나 입력 필드 내부 trailing icon은 장식일 수도 있지만, 음성 검색/스캐너/clear처럼 실제 액션이면 부모 field와 별도 affordance로 본다.
- 복합 row 안에 토글과 상세 이동이 함께 있으면 하나의 row 의미로 뭉개지지 않게 실제 trigger 분리를 우선 검토한다.

## Todo App Interpretation

`todo list` 앱을 별도 지시 없이 만들 때는 아래 방향을 기본 해석으로 사용한다.

- 목표는 복잡한 프로젝트 툴보다 빠른 정리와 완료감 제공
- 메인 객체는 task, section, routine, completed state
- 핵심 상호작용은 추가, 체크 완료, 우선순위 파악, 세부 수정
- 시각 톤은 지나치게 공격적이기보다 차분하고 반복 사용 가능한 방향 우선
- 첫 화면 비주얼은 실제 task row, check indicator, summary card 패턴을 활용

## 작성 시 체크 질문

- 이 화면의 1순위 행동이 3초 안에 보이는가?
- `app bar`가 현재 화면의 맥락을 설명하는가?
- `settings`는 장식보다 상태와 변경 가능성이 잘 보이는가?
- `endDrawer`는 보조 탐색 역할에 머물고 있는가?
- 이 화면을 다른 화면 옆에 두었을 때 같은 앱처럼 보이는가?
